# 22_顧客担当者編集（A04_17）

## 機能概要

既存の顧客担当者情報を編集する機能です。入力→確認→確定の3画面構成で、氏名、メールアドレス、連絡先、部署、主担当フラグを編集できます。メールアドレスの重複チェック（自ID除外、グローバル）を実施します。主担当フラグを設定した場合、同顧客の他の担当者の主担当フラグは自動的にOFFになります。パスワードは本機能では変更できません。

## 画面フロー

```
[担当者一覧画面]
    ↓ 編集ボタンをクリック
    ↓ GET /admin/contact/edit?id={担当者UUID}&customerId={顧客UUID}
    ↓ データベースから既存情報を取得
    ↓
[編集入力画面] - 現在の情報が初期表示
    ↓ 担当者情報を編集
    ↓ POST /admin/contact/edit_check
    |
    ├─ バリデーションエラー → [編集入力画面] へ戻る
    ├─ メール重複エラー → [編集入力画面] へ戻る
    └─ バリデーションOK → [編集確認画面] へ遷移
        ↓ 確定ボタンをクリック
        ↓ POST /admin/contact/edit_done
        |
        ├─ メール重複エラー → [編集入力画面] へ戻る
        ├─ システムエラー → [エラー画面] へリダイレクト
        └─ 更新成功 → [編集完了画面] を表示
```

### 処理フロー

#### 1. 編集フォーム表示（GET `/admin/contact/edit?id={UUID}&customerId={UUID}`）

1. **パラメータ取得・検証**
   - `id`: 編集対象の担当者ID（UUID形式）
   - `customerId`: 顧客ID（UUID形式）
   - `validation.isUuid(id)` および `validation.isUuid(customerId)` でUUID形式チェック
   - 不正な形式の場合: "不正なIDが指定されました。" エラー

2. **既存データ取得**
   - `CustomerContactDAO.selectById(id)` で担当者情報を取得
   - DTO → Domainオブジェクトに変換

3. **顧客情報取得**
   - `ensureCustomerOnRequest()` ヘルパーメソッドで顧客情報を取得
   - `customer` 属性に設定（ヘッダー表示用）

4. **フォーム表示**
   - `contact` 属性に既存データを設定
   - 編集ビュー（`contact/admin/edit`）へフォワード

#### 2. 編集確認（POST `/admin/contact/edit_check`）

1. **入力パラメータ取得**
   - `id`: 担当者ID（hidden）
   - `customerId`: 顧客ID（hidden）
   - `name`: 氏名（必須）
   - `mail`: メールアドレス（必須）
   - `phone`: 電話番号（任意）

2. **バリデーション（必須・形式チェック）**
   - 必須チェック:
     - `validation.isNull("ID", idStr)`
     - `validation.isNull("顧客ID", cidStr)`
     - `validation.isNull("氏名", name)`
     - `validation.isNull("メール", mail)`
   - 形式チェック:
     - `validation.isEmail(mail)` - メールアドレス形式
     - `validation.isPhoneNumber(phone)` - 電話番号形式（入力がある場合のみ）

3. **重複チェック（自ID除外、グローバル）**
   - バリデーションエラーがない場合のみ実施
   - `CustomerContactDAO.mailExistsExceptId(mail, id)` で自ID以外でメール重複確認
   - 重複時: "このメールアドレスは既に登録されています。" エラー

4. **エラー時の処理**
   - `errorMsg` 属性にエラーメッセージを設定
   - ID、顧客ID、入力値を属性に戻す（フォーム再表示用）
   - 顧客情報を再取得して設定
   - 編集入力画面へ戻る

5. **成功時の処理**
   - ID、顧客ID、入力値を属性に設定
   - 顧客情報を再取得して設定
   - 確認画面（`contact/admin/edit_check`）へフォワード

#### 3. 編集確定（POST `/admin/contact/edit_done`）

1. **入力パラメータ取得・バリデーション**
   - 確認画面と同じパラメータを再取得
   - 追加パラメータ:
     - `nameRuby`: ふりがな（任意）
     - `department`: 部署（任意）
     - `isPrimary`: 主担当フラグ（"true"/"false"）
   - 必須チェック:
     - `validation.isNull("ID", idStr)`
     - `validation.isNull("顧客ID", cidStr)`
     - `validation.isNull("氏名", name)`
     - `validation.isNull("メール", mail)`
   - 形式チェック（入力がある場合のみ）

2. **重複チェック（再実行）**
   - `CustomerContactDAO.mailExistsExceptId(mail, id)` で自ID以外でメール重複チェック
   - 重複時: エラーメッセージを設定して編集入力画面へ戻る

3. **エラー時の処理**
   - `errorMsg` 属性にエラーメッセージを設定
   - ID、顧客ID、入力値を属性に戻す（フォーム再表示用）
   - 顧客情報を再取得して設定
   - 編集入力画面へ戻る

4. **更新データ作成**
   - DTOオブジェクトに新しい値を設定:
     - `id`: 担当者ID（UUID）
     - `mail`: メールアドレス
     - `name`: 氏名
     - `nameRuby`: ふりがな
     - `phone`: 電話番号
     - `department`: 部署
     - `isPrimary`: 主担当フラグ（Boolean）

5. **データベース更新**
   - `CustomerContactDAO.update(dto)` で更新実行

6. **主担当フラグ処理**
   - 主担当フラグがONの場合:
     - `CustomerContactDAO.clearPrimaryForCustomer(customerId)` で同顧客の全担当者の主担当フラグをOFF
     - `CustomerContactDAO.setPrimaryById(id, true)` で自分の主担当フラグをON
   - 主担当フラグがOFFの場合:
     - `CustomerContactDAO.setPrimaryById(id, false)` で自分の主担当フラグをOFF
   - トランザクションコミット

7. **完了画面表示**
   - `message` 属性に "担当者の更新が完了しました（件数:1）。" を設定
   - 顧客情報を再取得して設定
   - 完了画面（`contact/admin/edit_done`）へフォワード

### トランザクション管理

```java
try (TransactionManager tm = new TransactionManager()) {
    UUID id = UUID.fromString(idStr);
    UUID customerId = UUID.fromString(cidStr);
    CustomerContactDAO dao = new CustomerContactDAO(tm.getConnection());

    if (dao.mailExistsExceptId(mail, id)) {
        validation.addErrorMsg("このメールアドレスは既に登録されています。");
        req.setAttribute(A_ERROR_MSG, validation.getErrorMsg());
        req.setAttribute(P_ID, idStr);
        req.setAttribute(P_CUSTOMER_ID, cidStr);
        pushFormBackToRequest();
        ensureCustomerOnRequest();
        return VIEW_EDIT;
    }

    CustomerContactDTO dto = new CustomerContactDTO();
    dto.setId(id);
    dto.setMail(mail);
    dto.setName(name);
    dto.setNameRuby(req.getParameter(P_NAME_RUBY));
    dto.setPhone(phone);
    dto.setDepartment(req.getParameter(P_DEPT));
    dto.setPrimary(isPrimary);

    int num = dao.update(dto);

    if (isPrimary) {
        dao.clearPrimaryForCustomer(customerId);
        dao.setPrimaryById(id, true);
    } else {
        dao.setPrimaryById(id, false);
    }

    tm.commit();
    req.setAttribute(A_MESSAGE, "担当者の更新が完了しました（件数:" + num + "）。");
    ensureCustomerOnRequest();
    return VIEW_EDIT_DN;

} catch (RuntimeException e) {
    validation.addErrorMsg("データベースに不正な操作が行われました");
    req.setAttribute(A_ERROR_MSG, validation.getErrorMsg());
    ensureCustomerOnRequest();
    return req.getContextPath() + req.getServletPath() + "/error";
}
```

### DAO呼び出し

- **CustomerDAO**
  - `selectByUUId(UUID id)`: 顧客情報を取得（ヘッダー表示用）
    - SQL: `SELECT c.* FROM customers c WHERE c.deleted_at IS NULL AND c.id = ?`

- **CustomerContactDAO**
  - `selectById(UUID id)`: 担当者情報を取得
    - SQL: `SELECT cc.* FROM customer_contacts cc WHERE cc.deleted_at IS NULL AND cc.id = ?`
  - `mailExistsExceptId(String mail, UUID excludeId)`: 自ID以外でメール重複チェック（グローバル）
    - SQL: `SELECT COUNT(*) FROM customer_contacts WHERE deleted_at IS NULL AND mail = ? AND id <> ?`
  - `update(CustomerContactDTO dto)`: 担当者情報を更新（パスワードは対象外）
    - SQL:
      ```sql
      UPDATE customer_contacts
      SET mail = ?, name = ?, name_ruby = ?, phone = ?, department = ?, is_primary = ?,
          updated_at = CURRENT_TIMESTAMP
      WHERE id = ?
      ```
  - `clearPrimaryForCustomer(UUID customerId)`: 指定顧客の全担当者の主担当フラグをOFF
    - SQL: `UPDATE customer_contacts SET is_primary = FALSE, updated_at = CURRENT_TIMESTAMP WHERE customer_id = ?`
  - `setPrimaryById(UUID contactId, boolean isPrimary)`: 指定担当者の主担当フラグを更新
    - SQL: `UPDATE customer_contacts SET is_primary = ?, updated_at = CURRENT_TIMESTAMP WHERE id = ?`

### セキュリティ

- **メール重複チェック**: 自分以外のアカウントとの重複を防止（全顧客の担当者でグローバルに重複チェック）
  - 確認画面と確定処理の両方で実施（二重チェック）
- **二重バリデーション**: 確認画面と確定処理で2回検証（改ざん対策）
- **UUID検証**: 不正なID形式をエラーとして処理
- **主担当正規化**: 1顧客につき1名のみ主担当として設定可能

## 備考

### 機能の仕様

- 更新日時（`updated_at`）は自動更新される
- 作成日時（`created_at`）、削除日時（`deleted_at`）は変更されない
- 最終ログイン日時（`last_login_at`）は変更されない
- パスワードは本機能では変更できない（担当者本人のマイページ編集で可能）
- 編集完了後、担当者一覧画面へ戻るリンクが表示される
- メールアドレス変更時、既存のパスワードは維持される

### 主担当フラグの仕組み

- **1顧客1主担当**: 1つの顧客につき主担当は1名のみ
- **主担当ON時の処理**:
  1. 同顧客の全担当者の主担当フラグをOFFにする（`clearPrimaryForCustomer`）
  2. 編集対象の担当者の主担当フラグをONにする（`setPrimaryById`）
- **主担当OFF時の処理**:
  1. 編集対象の担当者の主担当フラグをOFFにする（`setPrimaryById`）
  2. 他の担当者の主担当フラグは変更されない
- **トランザクション内で処理**: 主担当の正規化はトランザクション内で実施されるため、途中でエラーが発生した場合はロールバックされる

### メール重複チェックの範囲

- **グローバル重複チェック**: 全顧客の全担当者でメールアドレスの重複をチェック
- 他の顧客の担当者と同じメールアドレスは登録不可
- 自分自身のメールアドレスは除外（変更しない場合もOK）
- 秘書（secretaries）とは別管理（重複しても問題ない）

### その他

- 担当者ID、顧客IDは編集画面でhiddenフィールドとして引き継がれる
- 顧客情報ヘッダーは全画面で表示される（どの顧客の担当者を編集しているか明示）
- 電話番号は任意項目で、形式チェックのみ実施
- ふりがな、部署は任意項目で、形式チェックなし
- パスワード変更は別機能（担当者本人のマイページ編集）で実施
